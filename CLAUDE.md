## 0. 프로젝트 컨텍스트

**PHASE**는 사용자가 좋아하는 소리를 녹음해 모으고, 그 소리들이 generative 엔진을 통해
하나의 공간적인 앰비언트 음악이 되는 모바일 앱입니다. 단순한 사운드 도구가 아니라,
소리를 추억의 단위로 다루는 **앰비언트 다이어리**입니다.

- **플랫폼**: React Native, **Expo + Development Build** (Expo Go 아님 — 오디오 라이브러리가
  네이티브 커스텀 코드를 포함해 Expo Go에는 없음)
- **오디오**: react-native-audio-api (Web Audio 명세 호환). 사운드는 신스 합성이 아니라
  `AudioBufferSourceNode` 기반 **파일 재생**(기본 제공 사운드 + 사용자 녹음 사운드 모두 동일)
- **지도**: MapLibre (`@maplibre/maplibre-react-native`) — 지도 무드가 앱 정체성에 중요해
  react-native-maps 대신 선택. 다크/라이트 톤 커스텀 스타일 적용
- **시각화**: react-native-skia 또는 react-native-svg + Animated
- **상태관리/스토리지**: 아직 미정 — 작업 시작 전 확인

### 시그니처 컴포넌트 — Phase Orbit

위상이 어긋나는 루프들을 보여주는 시각화. **앱에서 가장 중요한 시각 요소이며, 절대 단순화
하거나 제거하지 않습니다.** 4레버(스케일/밀도/공간/템포)와 항상 같은 화면에 있어야 의미가
있습니다. 이 컴포넌트를 건드릴 때는 디자인 의도(위상이 서로 다른 속도로 도는 점들)를 반드시
보존해야 합니다.

### 디자인 토큰

- 배경: `#FAFAF8` (true white `#FFFFFF`나 웜 크림톤 아님 — 정확히 이 값 사용)
- 키 액센트: `#D97757` (주황 — 정밀한 강조용, 화면 전체에 남발하지 않음)
- 보조색: slate(텍스처), bone(퍼커시브) — 중성 그레이 계열, 웜 베이지 아님
- **폰트: 모노스페이스 고정** (SF Mono / JetBrains Mono / DM Mono 계열). Roboto 등 일반
  산세리프 사용 금지. 헤더/라벨은 letter-spacing을 살짝 넓게.
- 모서리: 10–16px 둥근 모서리, 헤어라인 보더 위주 (드롭섀도 지양)

### 폴더 구조

```
app/                 Expo Router 화면
components/          공유 UI (PhaseOrbit.tsx 포함, 격리해서 관리)
audio/               오디오 엔진 — UI와 분리 (engine/scheduler/phasing/recorder)
store/               Sound / Bank / Pad / GenerativePreset 모델
assets/builtin-sounds/  기본 제공 사운드 파일
theme/               colors.ts, typography.ts — 색상/폰트는 반드시 여기서만 정의
```

`audio/`는 화면(UI)과 독립적으로 동작해야 합니다. PhaseOrbit이 오디오 엔진의 시간 기준
(transport currentTime)을 구독하는 구조이므로, 이 시간 기준을 어떻게 노출할지가 오디오
포팅의 1순위 요구사항입니다.

### 화면 구성 (MVP)

1. 샘플러 (8패드 그리드, 뱅크 스와이프, 볼륨/mute/마스터)
2. 녹음/트림 (파형 + 트림 핸들)
3. 기록/메모 (선택 사항 — 건너뛰기 가능, 강제 아님)
4. 소리 지도 (커스텀 스타일 지도 + 핀, 클러스터링 없음)
5. 제너레이티브 (4레버 + Phase Orbit)

## 2. 코딩 전에 먼저 생각하기

**짐작하지 않습니다. 혼란을 숨기지 않습니다. 트레이드오프를 드러냅니다.**

구현 전에:

- 이해한 내용을 먼저 말하고 설명한 뒤, 진행 여부를 묻습니다.
- 가정한 것을 명시적으로 말합니다. 불확실하면 묻습니다.
- 여러 해석이 가능하면 전부 제시합니다 — 임의로 하나를 고르지 않습니다.
- 더 단순한 방법이 있으면 말합니다. 필요하면 반대 의견을 냅니다.
- 불명확한 부분이 있으면 멈춥니다. 무엇이 혼란스러운지 specify하고 묻습니다.

## 3. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드는 작성하지 않습니다.**

- 요청받지 않은 기능은 추가하지 않습니다.
- 한 번만 쓰는 코드에 추상화를 만들지 않습니다.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 넣지 않습니다.
- 일어날 수 없는 상황에 대한 에러 처리를 하지 않습니다.
- 200줄로 쓴 코드가 50줄이 될 수 있다면, 다시 씁니다.

스스로 물어봅니다: "시니어 엔지니어가 이걸 보고 과하게 복잡하다고 할까?" 그렇다면 단순화합니다.

## 4. 외과적 수정

**반드시 건드려야 하는 것만 건드립니다. 자신이 만든 흔적만 정리합니다.**

기존 코드를 수정할 때:

- 주변 코드, 주석, 포맷을 "개선"하지 않습니다.
- 망가지지 않은 것을 리팩토링하지 않습니다.
- 자신의 스타일과 다르더라도 기존 스타일을 따릅니다.
- 관련 없는 dead code를 발견하면 언급만 하고, 삭제하지 않습니다.

변경으로 인해 고아가 생기면:

- 자신의 변경으로 인해 더는 쓰이지 않게 된 import/변수/함수는 제거합니다.
- 자신의 변경과 무관한 기존 dead code는 요청받지 않았다면 제거하지 않습니다.

기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 합니다.

## 5. 목표 주도 실행

**성공 기준을 정의합니다. 검증될 때까지 반복합니다.**

작업을 검증 가능한 목표로 바꿉니다:

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시키기"
- "X 리팩토링" → "리팩토링 전후 테스트가 모두 통과하는지 확인"

여러 단계가 필요한 작업은 간단한 계획을 먼저 말합니다:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강한 성공 기준은 독립적으로 반복 작업을 할 수 있게 합니다. 약한 기준("작동하게 만들기")은
계속 추가 확인을 요구하게 만듭니다.
